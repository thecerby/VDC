VDC : Secure Source Code Statement

Created by Stephan Oomen on 15 mrt., 2022

This statement describes an overview how the software RegistratieXpert is secured on different aspects of the application and development lifecycle.

Principles

Each contributor needs to take the following into consideration:

  • Do not trust the user

  • Keep it secure and simple

  • Question yourself: Is there a security implication here?

  • OWASP Top 10

  • Security implications are discussed

Storage and access

The software is persisted into GIT version control and is stored in Azure DevOps Services.

Contributors are on invite-only and is being done through SSO of the company.

Pull Requests

In order to contribute to the software, a PR must be made targeting the main branch. Here the Lead developer needs to approve the PR and the Continues Integration build needs to pass all unit-tests.

Changes made to the software must include:

  • User Story ID, found in the Issue tracking System

  • Description of the change

  • Unit- and/or integration test.

  • Approval by the Lead developer

Whenever a Pull Request is approved and new changes are pushed, the review is reset.

External dependencies

Before introducing an external dependency, this will be reviewed by the Lead developer on the following topics:

  • History of security vulnerabilities

  • How are security vulnerabilities being handled.

  • Maintainability, versioning

  • Popularity

External dependencies (e.g. packages) used in the software is being checked regularly for risks found.
Dependencies will be updated with at-least the latest minor version.

Restricted sources of external dependencies:

All external sources are prohibited unless listed below:

  • NuGet official source

  • NPM official source

  • Company official source

  • Files added directly in source control

Secrets / credentials

All production secrets and/or credentials are being stored in a secure environment such as a Key Vault or Key Store. When a release is being made, these secrets can be read through the automated deployment pipeline to the corresponding environment.

Deployment

All deployments are done through Azure DevOps Services and is automated. Prior pushing builds to the environment, the people responsible for the environment needs to approve the deployment process prior starting the deployment. This approval is registered.
